# Layer 2 仕様書 — 記録する（年輪を刻む）

## 概要

Layer 2は「推し活の年輪」コンセプトの核心。
訪れた体験をファン人生の資産に変える層。

Layer 1（発見する）が「どこに行くか」を解決するのに対し、
Layer 2は「行った体験を残す」を解決する。

> 「2年前のあの日、インフルエンサーのロケ地に行った」が、ここに残っている

---

## 解決する課題

- 訪問チェックだけでは「消したら終わり」の設計になっている
- 日時・記憶・感情が残らないため、蓄積の価値が生まれていない
- アンインストールのコストがゼロ＝手放せないアプリになれていない
- チェックインが「作業」であり「体験」になっていない

---

## 機能一覧

### 1. 訪問日時の自動記録

- チェックイン時にタイムスタンプを自動保存する
- 後から編集可能にする（「実は昨日行った」に対応）
- 表示形式：`2025年10月14日`
- **訪問記録（チェックイン）自体の削除は行わない**（年輪は削れない）

### 2. メモ機能

- 任意入力・140文字以内
- 「ここで何を感じたか」を残すための一言メモ
- 入力しなくても訪問記録は成立する（任意）
- **後から編集・削除が可能**（訪問日時と同様）

### 3. チェックイン完了演出

チェックインの瞬間は「年輪が刻まれる」瞬間であり、Layer 2の核心アクション。
トースト通知だけでは御朱印帳の「押してもらった瞬間の満足感」に相当するものがない。
記録する行為自体が報酬になる体験を設計する。

#### チェックイン完了画面（fullScreenCover）

チェックイン完了時、トーストではなく**fullScreenCover（全画面モーダル）**を表示する。

「年輪が刻まれる瞬間」に相応しい重みを持たせるため、カード型モーダルではなく全画面を採用する。
メモ入力時のキーボード管理、誤タップによる閉じ防止の観点からもfullScreenCoverが適切（Apple HIG準拠）。

#### 完了演出の方針

静的なレイアウトだけでは「記録する行為自体が報酬になる体験」を十分に実現できない。
以下の演出を実装時に組み合わせ、御朱印帳で朱印を押してもらった瞬間の満足感を再現する。

- **Hapticフィードバック**（`.success`）— モーダル表示時に発火し、「刻んだ」感覚を手に残す
- **カウントバッジのアニメーション** — 「12回目の巡礼！」がスケールアップで弾むように登場
- **祝福アニメーション** — Lottie等を活用した紙吹雪や年輪モチーフ（同心円が広がる）の演出を検討

具体的な演出内容は実装フェーズで別Issueとして設計・検証する。

```
┌─────────────────────────────┐
│ 9:41                        │
│                             │
│      📍 乃木坂駅 1番出口     │
│                             │
│    ♪ 君の名は希望            │
│    24thシングル表題曲         │
│                             │
│    ── 2025年10月14日 ──      │
│    ✨ 12回目の巡礼！          │
│                             │
│    ☁️ 記録をクラウドに保存しました │
│                             │
│  ┌───────────────────────┐  │
│  │ メモを残す（任意）       │  │
│  └───────────────────────┘  │
│                             │
│  ┌───────────────────────┐  │
│  │       閉じる            │  │
│  └───────────────────────┘  │
│                             │
└─────────────────────────────┘
```

#### 仕様詳細

- **スポット名と作品名を大きく表示する**（「何の聖地に来たのか」を改めて実感させる）
- **作品種別に応じたアイコンを表示する**（SF Symbols使用）
  - 楽曲：`music.note`（例：`♪ 君の名は希望`）
  - 映像作品（番組/映画/特番）：`tv`（例：`📺 乃木坂工事中`）
  - 将来、雑誌など別媒体が増えた場合はアイコンを追加する
- **楽曲の場合、補足情報を表示する**（シングル/アルバム名、何枚目か）
  - 例：`24thシングル表題曲`、`アルバム「今が思い出になるまで」収録`
  - Layer 1のスポットマスターデータにメタ情報を持たせる（別タスク）
- **累計巡礼回数を日付の直下にバッジ表示する**（「12回目の巡礼！」）
  - 日付→回数の流れで「積み上がり」を即座に実感させる配置
  - **全スポット通算ののべ訪問回数**をカウントして表示する（同じスポットへの再訪問もカウントに含む）
  - 例：3つのスポットに計12回訪問した場合 →「12回目の巡礼！」
  - 初回チェックイン時は「最初の巡礼です🎉」と表示する
  - 意図：ユニークスポット数ではなくのべ回数を採用することで、再訪問でも数字が増え、巡礼頻度の高いAさん（週1巡礼）に毎回の報酬感を提供する
- **メモ入力欄をモーダル内に配置する**（その場で書ける・スキップもできる）
- **クラウド保存の旨をモーダル内に表示する**（トースト表示の代替）
- **「閉じる」ボタンは画面下部に配置する**（片手操作でのリーチャビリティ、メモ入力後の自然な操作フロー）
- fullScreenCoverのため、バックグラウンドタップでは閉じない

#### 機能追加ではなく体験設計の変更

従来のチェックインフロー：
```
チェックインボタン → トースト「記録を保存しました」→ 元の画面に戻る
```

改訂後のチェックインフロー：
```
チェックインボタン → 完了モーダル表示（楽曲情報・累計回数・メモ入力）→ 閉じて元の画面に戻る
```

トースト表示は廃止し、完了モーダルに統合する。

### 4. 巡礼手帳ビュー（新規画面）

訪問済みスポットを時系列で一覧表示する画面。
「自分の巡礼の年輪」が一目でわかる。

#### 巡礼サマリー（手帳冒頭）

巡礼手帳ビューの冒頭に、巡礼の全体像がわかるサマリーを常時表示する。
記録が増えるほど数値が育ち、「こんなに回ったんだ」という実感を生む。

```
┌─────────────────────────────┐
│  巡礼済み        総巡礼回数   │
│    12              28       │
│    箇所             回       │
└─────────────────────────────┘
```

##### サマリーの表示項目

| 項目 | 算出ロジック | 表示例 |
|---|---|---|
| 巡礼済み箇所数 | 訪問済みユニークスポット数 | `12 箇所` |
| 総巡礼回数 | 全スポット通算ののべ訪問回数（再訪問含む） | `28 回` |

- 訪問記録が0件の場合、サマリーは非表示（空状態のデザインが優先）
- 訪問記録が1件以上になった時点で表示を開始する

##### 旧項目からの変更理由

- **楽曲数を削除**: スポットには楽曲だけでなく映像作品（ドラマ・番組）も含まれるため、「楽曲数」では実態と合わない
- **総巡礼回数を追加**: のべ回数は再訪問でも増え続け、「年輪が積み上がる」コンセプトに直結する。巡礼頻度の高いAさんに毎回の成長実感を提供
- **最初の巡礼日を削除**: 巡礼を始めたばかりのユーザーには意味が薄く、サマリーの情報密度を下げるだけになる

#### 表示構造（通常時）

作品名を視覚的に目立たせ、「乃木坂の聖地を巡っている」文脈を強調する。
月単位のセクションは折りたたみ可能とし、記録が100件を超えても快適にナビゲーションできるようにする。

```
┌─────────────────────────────┐
│  巡礼済み        総巡礼回数   │
│    12              28       │
│    箇所             回       │
└─────────────────────────────┘

▼ 2025年10月（2件）  ← 展開状態（タップで折りたたみ）

  📍 乃木坂駅 1番出口
     ♪ 君の名は希望
     10.14
     「発車メロディ流れてた、泣けた」

  📍 国立新美術館
     ♪ それぞれの椅子
     10.14

▶ 2025年8月（1件）   ← 折りたたみ状態（タップで展開）

▶ 2025年5月（3件）

▶ 2024年12月（2件）

▶ 2024年8月（1件）
```

##### 月セクションの折りたたみ仕様

- 直近の月は**展開状態**で表示する
- 過去の月は**折りたたみ状態**で表示する（タップで展開）
- 展開/折りたたみの状態はセッション中のみ保持する（永続化しない）
- 「ファン人生の年表」として、時間軸で記録を俯瞰できる構造を実現する

#### 空状態（訪問記録0件）の表示

初めて巡礼手帳ビューを開いたとき、訪問記録が0件の状態で離脱されると年輪が始まらない。
Cさん（新規ファン）の「始めやすさが大事」という期待に応えるため、空状態を明示的に設計する。

```
    🗾

  まだ巡礼の記録がありません

  聖地を訪れてチェックインすると
  ここに記録が積み上がっていきます

  ［近くの聖地を探す］  ← タップでスポット一覧へ遷移
```

- 「何もない」ではなく「これから始まる」と感じさせるコピーにする（細かい言い回しは実装時に調整）
- CTAボタンでスポット一覧（Layer 1）に直接誘導する
- 説明文は短く、操作の次の一手が明確になるようにする
- 空状態ではサマリーもクラウド表示も非表示（画面をシンプルに保ち、最初の一歩に集中させる）

#### 仕様詳細（通常時）

- **月単位でグルーピングし、折りたたみ可能にする**（直近月は展開、過去月は折りたたみ）
- 新しい訪問順（降順）で表示する
- メモがある場合のみメモ行を表示する
- 未訪問スポットは表示しない
- **作品名を作品種別アイコン付きで各スポットの直下に表示する**（楽曲：`music.note`、映像作品：`tv`）
- **巡礼サマリーを手帳ビュー冒頭に常時表示する**（空状態では非表示）
- **クラウド保存メッセージは巡礼手帳ビューには表示しない**（チェックイン完了モーダルとオンボーディングの2箇所で明示）

### 5. データの永続化・クラウド保存

#### ハイブリッド保存方針

データの性質に応じて保存戦略を分け、信頼性とシンプルさを両立する。

| データ種別 | ローカル保存 | リモート保存 | 通常時のソースオブトゥルース |
|---|---|---|---|
| スポットマスター（読み取り専用） | Firestoreキャッシュ（SDK自動管理） | Firestore | Firestore |
| 巡礼記録（ユーザーの書き込み） | SwiftData | Firestore | SwiftData |

> **巡礼記録のソースオブトゥルースの補足**
>
> 通常利用時はSwiftDataが正（オフライン書き込み・即時読み取りを保証）。
> ただしアプリ再インストール・機種変更時はSwiftDataが空になるため、
> サインイン後にFirestoreから復元する。
> SwiftDataは「即時の信頼性」、Firestoreは「長期の永続性」を担う相互補完の関係。
>
> | 状態 | SwiftData | Firestore | 正とするデータ |
> |---|---|---|---|
> | 通常利用 | あり | 同期済み | SwiftData |
> | オフライン中 | あり | 未同期 | SwiftData |
> | アプリ再インストール+サインイン | 空 | あり | Firestoreから復元 |

##### スポットマスター → Firestoreキャッシュのみ

サーバー管理のマスターデータであり、自前のローカルストアは持たない。
キャッシュが効いている間はオフラインでも読める。消えても再取得すれば済む。
起動時やフォアグラウンド復帰時に全スポットを取得しておけばキャッシュに載り続ける。

##### 巡礼記録 → SwiftData + Firestore同期

「刻んだはずの年輪が消えた」は絶対に起こしてはならない。
Firestoreキャッシュには構造的な弱点がある:

- キャッシュは「以前取得・書き込みしたデータの残留」であり、永続性が保証されていない
- アプリ再インストールやキャッシュクリアで消える
- LRUで予期せず古い記録が消える可能性がゼロではない
- オフライン書き込みのpending queueはアプリが強制終了されると失われることがある

巡礼記録はSwiftDataに永続保存し、Firestoreへの同期はクラウドバックアップとして行う。
SwiftDataモデルは軽量（スポットID・訪問日時・メモ・同期状態）で実装コストも小さい。

#### ローカルファースト方針

- チェックインは**常にSwiftDataに成功する**（オフラインでも記録される）
- ネットワーク復帰時にFirestoreへ自動同期する
- 完了モーダルの表示はネットワーク状態に応じて分岐する：

| ネットワーク状態 | モーダル表示 |
|---|---|
| オンライン | 「☁️ 記録をクラウドに保存しました」 |
| オフライン | 「📱 記録を端末に保存しました（オンライン時に自動で同期します）」 |

- 巡礼手帳ビューでは未同期の記録に同期状態を表示する：

| 状態 | 表示 |
|---|---|
| 同期済み | 表示なし（通常表示） |
| 未同期 | 「📱 同期待ち」をスポット名の横に表示 |

#### 競合解決方針

巡礼記録はcreate-onceのデータ（チェックイン時に作成、以降はメモ編集程度）であり、
単一端末利用の前提では同一レコードの同時編集が発生しないため、**Last Write Wins（LWW）** を採用する。

- ローカル書き込み時は端末時刻で `updatedAt` を記録する
- Firestore同期時は `FieldValue.serverTimestamp()` で `serverUpdatedAt` を別途打つ
- MVP（単一端末）では `updatedAt` のみで判定する
- マルチデバイス対応時には `serverUpdatedAt` で判定に切り替える（端末時刻のズレを回避）

##### 優先度

競合解決の精度よりも、**同期が確実に完了すること**の方が重要。
リトライの確実性や `syncStatus` の状態遷移（pending → synced）が漏れない設計に注力する。

#### クラウド保存の基本方針

- 訪問記録はクラウドに保存する
- アプリを削除しても記録が消えない
- ユーザーが安心して記録を積み上げられる心理的前提を作る

#### クラウド保存の明示場所

以下の2箇所で明示する。

| 場所 | 表示内容 | タイミング |
|---|---|---|
| オンボーディング（Screen 4） | 「記録はクラウドに安全に保存されます。アプリを削除しても消えません。」 | 初回起動時 |
| チェックイン完了モーダル | 「☁️ 記録をクラウドに保存しました」（オフライン時は文言分岐） | チェックイン直後 |

## スポット写真の扱いについて

スポット写真（Google Place API）はLayer 1のスポット詳細画面で既に実装済み。
Layer 2では写真の新規実装は行わない。

ただし以下の制約をプロダクト全体の方針として改めて明記する。

- **メンバーの顔写真は一切使用しない**
- **ユーザー投稿写真は受け付けない**

将来フェーズでユーザー写真の需要が明確になった場合に再検討する。

## 作品メタ情報の整備について

チェックイン完了モーダルで作品の補足情報を表示するため、
スポットマスターデータに作品メタ情報を追加する必要がある。

**これはLayer 2と並行して別タスクとして管理する。**

| データ項目 | 例 | 用途 |
|---|---|---|
| 作品種別 | 楽曲 / 映像作品 | アイコン切り替え（`music.note` / `tv`） |
| リリース種別 | シングル表題曲 / カップリング / アルバム曲 | 完了モーダル表示（楽曲の場合のみ） |
| リリース番号・名称 | 24thシングル / アルバム「今が思い出になるまで」 | 完了モーダル表示（楽曲の場合のみ） |

- Layer 2の完了条件には含めない
- メタ情報が未整備の場合のフォールバック表示は以下の通り：

| 状態 | モーダル表示 |
|---|---|
| 楽曲 + メタ情報あり | スポット名 → ♪ 作品名 → 補足情報（例：`24thシングル表題曲`）→ 日時 → 累計回数 |
| 楽曲 + メタ情報なし | スポット名 → ♪ 作品名 → 日時 → 累計回数（**補足情報行を非表示**） |
| 映像作品 | スポット名 → 📺 作品名 → 日時 → 累計回数（**補足情報行なし**） |
| 作品種別未設定 | スポット名 → 作品名（アイコンなし）→ 日時 → 累計回数 |

- 将来、雑誌など新しい媒体が増えた場合はアイコンと作品種別を追加する
- 作品メタ情報はLayer 3（達成する）でも活用する想定

## 認証・データ移行

### 認証方式

**Sign in with Apple（Firebase Auth経由）** を採用する。

- Firebase Authと既存Firestoreが直結し、実装コストが最小
- Apple IDに紐づくため、アプリ削除・機種変更でも記録を復元可能
- 将来SNS認証を追加する場合も、Firebase Authで既存アカウントにプロバイダを追加できる

### 既存ユーザーのデータ移行

現在はデバイスUUIDでチェックイン状態をFirestoreで管理している。
Layer 2リリース時に、UUIDベースからFirebase Auth IDベースへ移行する。

#### 移行フロー

```
アプリ更新後の初回起動
  → サインインを促す画面を表示（スキップ可）
  → 「記録を守るためにサインインしませんか？」
  → スキップした場合、従来通りUUIDで動作

次回チェックイン時
  → サインイン必須（チェックインの前に完了させる）
  → 「記録をクラウドに安全に保存するためにサインインが必要です」

サインイン成功
  → デバイスUUIDに紐づく既存データをFirebase Auth IDに紐づけ直す
  → 以降はFirebase Auth IDで管理
```

- 初回起動時はスキップ可にし、チェックイン時に必須化する二段構え
- 現場で焦らず、自宅でサインインを済ませる余地を残す
- サインインの理由を「記録を守るため」として、価値の一部として伝える

#### Remote Configによる移行制御

移行のペースをアプリアップデートなしで動的に調整するため、Firebase Remote Configで以下のフラグを管理する。

| フラグ | 型 | 説明 | 初期値 |
|---|---|---|---|
| `can_dismiss_signin_prompt` | Bool | サインイン促進をスキップできるか | `true` |
| `is_signin_required_for_checkin` | Bool | チェックイン時にサインインが必須か | `true` |
| `is_uuid_access_enabled` | Bool | UUIDベースのデータアクセスが有効か | `true` |

##### 移行フェーズの運用イメージ

```
Phase 1（リリース直後）
  can_dismiss_signin_prompt: true
  is_signin_required_for_checkin: true
  is_uuid_access_enabled: true
  → 仕様通りの二段構え

Phase 2（移行率が十分に上がったら）
  can_dismiss_signin_prompt: false
  is_signin_required_for_checkin: true
  is_uuid_access_enabled: true
  → まだ移行していないユーザーにサインインを強制

Phase 3（UUID廃止）
  can_dismiss_signin_prompt: false
  is_signin_required_for_checkin: true
  is_uuid_access_enabled: false
  → UUID経由のアクセスを完全停止
```

- 移行率が想定より低ければPhase 1に留まり、順調ならPhase 3まで進める
- 具体的なフェーズ移行のタイミングはアプリ更新率・移行率の運用データを見て決定する

#### UUID管理の期限

- Firestore上のUUIDデータは削除しない（ストレージコストは微小）
- アプリ側のUUID経由のアクセスは `is_uuid_access_enabled` フラグで制御する
- フラグ無効化後もFirestore上にデータは残るため、サポート対応での紐づけは技術的に可能

### アカウント削除

App Store審査ガイドライン（5.1.1(v)）により、アカウント作成機能があるアプリは**アカウントとデータの削除機能**の提供が必須。

#### 「年輪は削れない」との区別

| 操作 | 可否 | 理由 |
|---|---|---|
| 個別チェックインの削除 | **不可** | 年輪は削れない。誤操作は日時編集で対応 |
| アカウント全削除 | **必須** | App Store審査要件。ユーザーの権利として提供する |

この2つは異なる概念であり、仕様上も実装上も明確に区別する。

#### 削除時の処理対象

アカウント削除が実行された場合、以下をすべて削除・失効させる。

| 対象 | 処理 |
|---|---|
| Firestore上の訪問記録（日時・メモ） | 削除 |
| Firestore上のユーザープロフィール | 削除 |
| Firebase Authアカウント | 削除 |
| Appleトークン | 失効（Apple REST APIで revoke） |

#### 削除フロー

```
設定画面 →「アカウントを削除」
  → 確認ダイアログ：「すべての巡礼記録が完全に削除されます。この操作は取り消せません。」
  → 再確認（Apple ID認証）
  → 削除実行 → サインアウト → 初期画面に戻る
```

- 削除は**取り消せない**旨を明示する
- 再確認でApple ID認証を求め、誤操作を防止する
- 削除完了後は未認証状態に戻る

#### スコープ外

- 同一ユーザーの複数デバイス利用はレアケースのため、Layer 2では考慮しない

## オンボーディングとの関係

オンボーディング（初回起動フロー）はLayer 2と並行して設計・実装するが、
**別タスクとして管理する。**

Layer 2完了条件にオンボーディングは含まない。
ただしLayer 2リリース前にオンボーディングが完成していることを必須とする。
```
Layer 2実装 ──────────┐
Sign in with Apple導入 ┤
既存データ移行実装 ────┤
楽曲メタ情報整備 ──────┤
Analyticsイベント実装 ─┤
                      ├──→ 同時リリース
オンボーディング実装 ──┘
```

## データ設計方針

技術的な詳細はアーキテクチャ方針に従う。
ここではプロダクト観点での制約のみ定義する。

- 訪問記録はユーザー個人に紐づく（他ユーザーとは共有しない）
- ユーザー識別はFirebase Auth ID（Sign in with Apple）を使用する
- スポットのマスターデータ（名称・座標・楽曲情報）はFirestoreで管理し、アプリ側はFirestore SDKのキャッシュで読み取る（自前のローカルストアは持たない）
- 訪問記録（日時・メモ・同期状態）はSwiftDataに永続保存し、Firestoreへ自動同期する
- 通常利用時のソースオブトゥルースはSwiftData、再インストール・機種変更時はFirestoreから復元する（詳細は「ハイブリッド保存方針」を参照）
- スポットマスターのソースオブトゥルースはFirestore
- 楽曲メタ情報（リリース種別・番号）はスポットマスターデータに追加する

## 完了条件

以下がすべて満たされた状態をLayer 2完了とする。

- [ ] チェックイン時に訪問日時が自動保存される
- [ ] 訪問日時を後から編集できる
- [ ] メモを入力・保存・編集・削除できる
- [ ] チェックイン完了時にfullScreenCoverモーダルが表示される（スポット名・作品名・日時・累計回数）
- [ ] チェックイン完了モーダルからメモを入力できる
- [ ] チェックイン完了モーダルにクラウド保存の旨が表示される
- [ ] 巡礼手帳ビューの冒頭に巡礼サマリー（巡礼済み箇所数・総巡礼回数）が表示される
- [ ] 巡礼手帳ビューに訪問済みスポットが月単位の折りたたみセクションで時系列表示される（作品名付き）
- [ ] 巡礼手帳ビューの空状態（0件）が定義通りに表示され、スポット一覧へ遷移できる
- [ ] オフライン時もチェックインがローカルに成功し、オンライン復帰時に自動同期される
- [ ] チェックイン完了モーダルの文言がネットワーク状態に応じて分岐する
- [ ] 巡礼手帳ビューで未同期の記録に同期状態が表示される
- [ ] Sign in with Appleでサインインできる
- [ ] サインイン後、既存のUUIDデータがFirebase Auth IDに移行される
- [ ] 記録がクラウドに保存され、再インストール・機種変更後も復元される
- [ ] アカウント削除時にFirestore・Firebase Auth・Appleトークンがすべて削除・失効される
- [ ] アカウント削除前に確認ダイアログと再認証が表示される

## Layer 3 への接続

Layer 2で蓄積した訪問記録データは、Layer 3（達成する）の土台になる。

- 楽曲別バッジ → 訪問記録 × 楽曲データで算出
- 推しメン別コンプリート率 → 訪問記録 × メンバーデータで算出
- 楽曲メタ情報（リリース種別・番号）→ Layer 3の達成表示で活用

Layer 2が完成していない状態でLayer 3を実装しない。
