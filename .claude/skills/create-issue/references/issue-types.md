# Issue種別ガイド

Issue の種別に応じた書き方の指針。
SKILL.md の Step 1 で種別を判断し、Step 2 でテンプレートを埋める際に参照する。

種別はリポジトリのラベルから判断する（SKILL.md Step 3 参照）。
以下は種別ごとの書き方ポイント。

---

## バグ

- **概要**: 「〜すると〜になる」の形式で再現手順を含める
- **背景**: 期待する動作（本来どうあるべきか）を明記する
- **対応内容**: 原因が判明していれば記載、不明なら「要調査」
- **AC**: 再現手順で正しく動作すること + 既存機能が壊れないこと

```markdown
## 概要
マップ画面を開いた際、初期表示位置が乃木坂駅ではなくApple Park（デフォルト座標）になる。

## 背景
起動時は乃木坂駅を中心にマップが表示される想定だが、
MKMapView の初期化タイミングの問題でデフォルト座標が表示されている。

## Acceptance Criteria
- [ ] マップ画面を開いた際、乃木坂駅を中心とした地図が表示されること
- [ ] ピンをタップした後のマップ位置と初期表示位置が一致すること
```

## 機能追加

- **概要**: ユーザー視点で何ができるようになるか
- **背景**: なぜその機能が必要か（ユーザー課題）
- **対応内容**: 主要な実装方針。UI/データフロー/画面遷移等
- **AC**: ユーザー操作ベースで検証可能な条件

## 改善

- **概要**: 何がどう良くなるか
- **背景**: 現状の問題点
- **対応内容**: 改善アプローチ
- **AC**: Before/After が区別できる条件

## リファクタ / 技術的負債

- **概要**: どのコードを何の目的で整理するか
- **背景**: 現状の技術的問題（可読性・責務の肥大化・テスト困難等）。技術的負債の場合は放置した場合のリスクも明記する
- **対応内容**: リファクタの方針（分割・抽出・移動等）
- **AC**: 振る舞いが変わらないことを含める

```markdown
## Acceptance Criteria
- [ ] ClusterMapView の Coordinator が独立ファイルに分離されること
- [ ] リファクタ前後でマップの表示・操作に差異がないこと
- [ ] ビルドが通ること
```

## 設定・環境

- **対応内容**: 具体的な設定変更・ファイル変更を箇条書きで
- **AC**: 設定が正しく反映されていることを検証する条件
