# Acceptance Criteria 書き方ガイド

code-review スキルの「Issue要件チェック」で検証されることを前提に、
明確で検証可能な AC を書くための指針。

---

## 基本ルール

1. **チェックボックス形式**: 必ず `- [ ]` で書く
2. **動詞で始める**: 「〜されること」「〜しないこと」の形式
3. **1項目1検証**: 1つのチェックボックスで検証することは1つだけ
4. **曖昧語の禁止**: 「適切に」「うまく」「正しく」「きれいに」は使わない

---

## 良い例 / 悪い例

### 良い例
```markdown
- [ ] マップ画面を開いた際、乃木坂駅を中心とした地図が表示されること
- [ ] お気に入りボタンをタップすると、ハートアイコンが塗りつぶし状態に変わること
- [ ] オフライン時にチェックインを試みた場合、エラーアラートが表示されること
- [ ] リファクタ前後でマップの表示・操作に差異がないこと
```

### 悪い例
```markdown
- [ ] マップが正しく表示されること          ← 「正しく」が曖昧
- [ ] UIが適切に更新されること              ← 「適切に」が曖昧
- [ ] パフォーマンスが改善されること        ← 基準が不明
- [ ] エラーハンドリングが実装されること    ← 何のエラーで何が起きるか不明
```

---

## AC に含めるべき観点

### 正常系
- ユーザー操作に対する期待される結果

### 異常系・エラーケース
- ネットワークエラー時の挙動
- 不正な入力に対する挙動
- 権限が拒否された場合の挙動（位置情報等）

### 非機能要件（該当する場合）
- 既存機能への影響がないこと（リファクタ時）
- ビルドが通ること（リファクタ・設定変更時）
- 特定の OS バージョンでの動作（iOS 17.5 以上）

---

## AC の粒度

- **粗すぎる**: 「チェックイン機能が動作すること」→ 何をもって動作とするか不明
- **細かすぎる**: 「ボタンの角丸が8pxであること」→ 実装の詳細であり AC ではない
- **ちょうどよい**: 「チェックイン成功時にスタンプ画面に遷移すること」→ ユーザー操作と結果が明確
