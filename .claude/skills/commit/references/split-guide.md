# コミット分割の判断基準

## 基本原則

**1コミット = 1つの理由で変更された、独立してレビュー・revertできる単位**

「このコミットをrevertしたら、それ以外の変更に影響が出るか？」を問いかける。
影響が出るなら分割を検討する。

---

## 分割すべきケース

### 種別（type）が異なる変更が混在している

```
❌ 1コミットにまとめない:
  - LoginViewModel のバグ修正（fix）
  - ユニットテストの追加（test）

✅ 分割する:
  fix: ログイン失敗時のエラー状態が更新されない問題を修正
  test: ログインエラー系のユニットテストを追加
```

### 関係のない複数のIssueにまたがる変更

```
❌ 1コミットにまとめない:
  - Issue #42 のバグ修正
  - Issue #55 の新機能追加

✅ Issueごとに分割する
```

### リファクタと機能変更が混在している

リファクタリングは振る舞いを変えないことが前提。
機能変更を同じコミットに入れると、revert時に意図しない影響が出る。

```
✅ 分割する:
  refactor: LoginRepositoryの依存注入をDIコンテナ経由に変更
  feat: ログイン履歴の保存機能を追加
```

---

## まとめてよいケース

### 同一の目的で変更されたファイル群

```
✅ 1コミットでよい:
  feat: プロフィール編集画面を追加
  対象: ProfileEditView.swift, ProfileEditViewModel.swift
```

### 小さな付随変更（数行以内）

```
✅ 1コミットでよい:
  fix: タイムラインの日付フォーマットを修正
  対象: TimelineCell.swift（本体修正）+
        DateFormatter+Extension.swift（フォーマット定数を追加・数行）
```

### テストと実装が強く結合している場合

TDD等でテストと実装を同時に書いた場合は、セットでコミットしてよい。

```
✅ 1コミットでよい:
  feat: 記事のブックマーク保存ロジックを追加
  対象: BookmarkRepository.swift + BookmarkRepositoryTests.swift
```

## WIPコミットの扱い

作業中に `WIP:` や `tmp:` プレフィックスのコミットを作る場合、
Push する前に必ず整理する。

```bash
# 直近のWIPコミットを1つ前のコミットに統合
git commit --amend

# 複数のWIPコミットをまとめてつぶす
git rebase -i HEAD~<コミット数>
# → squash / fixup でまとめる
```

`WIP:` コミットがリモートブランチに Push されていないことを確認する。
