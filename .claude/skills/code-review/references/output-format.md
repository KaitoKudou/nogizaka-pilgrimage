# レポート出力フォーマット

コードレビュー Step 4 で使用する出力テンプレートと注意事項。

---

## 指摘の種別ラベル

各指摘には以下のラベルを先頭に付与し、対応の優先度を明確にする:

| ラベル | 意味 | 対応 |
|--------|------|------|
| **`[Must]`** | 必ず変更してほしい | バグ・クラッシュ・要件未達・セキュリティ等。マージ前に修正必須 |
| **`[Imo]`** | 必須ではないが、より良くなる | 設計改善・可読性向上・パフォーマンス等。著者の判断で採否を決めてよい |
| **`[Nits]`** | 軽微な指摘 | typo・命名の微調整・フォーマット等。余裕があれば対応 |
| **`[FYI]`** | 参考情報 | 対応不要。関連知識や代替案の共有 |

---

## テンプレート

- 指摘がないセクションは「特になし」と一行で記載し省略しない。
- 指摘事項は重要度順（Must → Imo → Nits → FYI）に並べる。

```markdown
## 📋 Issue要件チェック
<!-- Issueがない場合はこのセクションごと省略 -->

| # | 要件 | 状態 | 備考 |
|---|------|------|------|
| 1 | ログイン後にホーム画面へ遷移する | ✅ | `LoginViewModel.swift:58` で対応 |
| 2 | エラー時にアラート表示 | ❌ | 実装が見当たらない |
| 3 | オフライン時の挙動 | ⚠️ | ネットワークエラーのハンドリングが未実装 |

---

## 指摘事項

### [Must] [ファイル名:行番号] タイトル
**問題**: 何が問題か
**理由**: なぜ問題なのか
**提案**:
\`\`\`swift
// 改善後のコード例（短い場合のみ）
\`\`\`

### [Imo] [ファイル名:行番号] タイトル
**現状**: 現在のコード/設計の説明
**提案**: どう改善できるか

### [Nits] [ファイル名:行番号] タイトル
**指摘**: typo や命名の改善案

### [FYI] タイトル
**情報**: 関連する知識や代替アプローチの共有

---

## 🟢 良い点

良いプラクティスや、特筆すべき実装の工夫。
レビューはネガティブなフィードバックだけでなく、良い点も明示する。

---

## 📊 サマリー

| 観点 | 評価 | コメント |
|------|------|---------|
| Issue要件 | 3/4達成 | エラー表示の実装が未達成 |
| 可読性 | ✅ 良好 | 命名・構造ともに明確 |
| 保守性 | ⚠️ 要注意 | DI未使用でテストが困難 |

| ラベル | 件数 |
|--------|------|
| Must | 1 |
| Imo | 2 |
| Nits | 1 |
| FYI | 0 |
```

---

## 注意事項

- **指摘の粒度**: 同種の問題は代表例1〜2件を挙げ、「他にも同様の箇所あり」と添える。全件列挙はしない。
- **トーン**: 断定的な否定表現（「〜は間違い」）より、提案・改善の言葉を使う。
- **不確かな箇所**: プロジェクトの背景が不明で判断できない指摘は、「〜と思われるが、背景によっては適切な場合もある」と留保をつける。
