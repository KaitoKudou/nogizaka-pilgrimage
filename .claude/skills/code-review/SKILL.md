# Code Review Skill

iOS/Swiftプロジェクトを対象としたコードレビュースキル。
以下の3つの観点でコードを評価し、構造化されたレポートを出力する。

1. **Issue要件チェック** — Issueに書かれた要件・ACが実装で満たされているか
2. **可読性** — 命名・構造・イディオムが読み手に優しいか
3. **保守性** — アーキテクチャ整合性・テスタビリティ・将来の変更耐性

---

## Step 1: 入力情報の収集

### 1-1. Issueの取得

**GitHubのIssue番号が指定された場合**（例: `#42`）:
```bash
gh issue view <番号> --json title,body,labels
```
取得できない場合はユーザーにIssueの内容をテキストで貼ってもらう。

**テキストで渡された場合**:
そのまま使用する。

Issueが存在しない場合は、Issue要件チェックのセクションをスキップし、
その旨をレポートに明記する。

### 1-2. レビュー対象コードの取得

以下の優先順位で対象コードを決定する:

1. **ユーザーがファイルやコードを直接貼った場合** → そのまま使用
2. **PR番号が指定された場合**（例: `#169`）:
   ```bash
   gh pr diff <番号>
   ```
3. **ブランチ名が指定された場合**:
   ```bash
   git diff main...<ブランチ名>
   ```
4. **何も指定がない場合** → ユーザーに確認する:
   - 直近のコミットとの差分？ (`git diff HEAD~1`)
   - 特定のブランチとの差分？
   - 特定のファイル？

差分だけでなく、変更ファイルの全体を読み、周辺コードとの整合性も確認する。

---

## Step 2: Issue要件の解析

Issueのテキストから以下を抽出する:

- **機能要件**: 実装すべき振る舞い
- **AC（Acceptance Criteria）**: 明示的なチェック項目
- **非機能要件**: パフォーマンス・アクセシビリティ・エラーハンドリング等の言及

ACが明記されていない場合は、Issue本文から暗黙の要件を推測してリスト化する。
その際、「推測」であることをレポートに明記する。

---

## Step 3: コードレビューの実施

### 3-1. Issue要件チェック

抽出した要件・ACに対し、コードで対応している箇所を探す。
対応箇所が見つからない、または不完全な場合は未達成とする。

チェック項目ごとに:
- ✅ 達成: どのファイル・関数で対応しているかを示す
- ❌ 未達成: 実装が見当たらない旨と、対応が必要な理由を簡潔に述べる
- ⚠️ 部分的: 実装はあるが不完全な点を指摘する

### 3-2. 可読性レビュー

命名・構造・コメント・Swiftイディオムの観点で評価する。
詳細なチェックリストは `references/swift-readability.md` を参照。

### 3-3. 保守性レビュー

アーキテクチャ整合性・DI・リソースアクセス・Concurrency 等の観点で評価する。
詳細なチェックリストは `references/architecture-patterns.md` を参照。

---

## Step 4: レポートの出力

`references/output-format.md` のテンプレートに従って出力する。

各指摘には種別ラベル（`[Must]` / `[Imo]` / `[Nits]` / `[FYI]`）を付与する。
ラベルの定義・テンプレート・注意事項の詳細は同ファイルを参照。
